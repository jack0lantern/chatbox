## 1. NextAuth Spotify OAuth

### Changes to `server/lib/auth.ts`

- Add `PrismaAdapter` from `@auth/prisma-adapter` — stores OAuth tokens in the `Account` table
- Add `SpotifyProvider` from `next-auth/providers/spotify`
- Switch session strategy from `jwt` to `database` (required for PrismaAdapter to link OAuth accounts to users)
- Add `callbacks.session` to expose `userId` on the session object

### Environment Variables

```
SPOTIFY_CLIENT_ID=<from Spotify Developer Dashboard>
SPOTIFY_CLIENT_SECRET=<from Spotify Developer Dashboard>
```

### OAuth Scopes

```
playlist-modify-public
playlist-read-private
user-read-email
```

`user-read-email` is required by NextAuth for user identification. 30-second preview URLs are included in track objects by default — no additional scope needed.

### Callback URL

```
http://localhost:3000/api/auth/callback/spotify
```

Set this in the Spotify Developer Dashboard under Redirect URIs.

### Token Storage

Spotify access tokens and refresh tokens are stored in the `Account` table (NextAuth standard). The `Account` model already has `access_token`, `refresh_token`, `expires_at`, `token_type`, and `scope` fields — no schema changes needed.

### Credential Provider Coexistence

The existing `CredentialsProvider` (email/password) remains for platform login. Spotify is an **account linking** provider — users log in with credentials, then connect Spotify from within the app. This requires configuring NextAuth to allow linking multiple providers to a single user account.

## 2. Spotify API Proxy

### Route: `POST /api/plugins/proxy/spotify`

A server-side proxy for all Spotify Web API calls. The iframe never communicates with Spotify directly.

### Request Format

```json
{
  "action": "search_songs" | "create_playlist" | "add_to_playlist",
  "params": { ... }
}
```

### Actions

**`search_songs`**
- Params: `{ query: string }`
- Proxies to: `GET https://api.spotify.com/v1/search?type=track&q={query}&limit=20`
- Returns: Array of track objects with `id`, `name`, `artist`, `album`, `albumArt` (image URL), `previewUrl` (30s clip), `duration`, `uri`

**`create_playlist`**
- Params: `{ playlistName: string, songs: string[] }` (songs are Spotify track URIs)
- Proxies to:
  1. `GET https://api.spotify.com/v1/me` — get Spotify user ID
  2. `POST https://api.spotify.com/v1/users/{userId}/playlists` — create playlist
  3. `POST https://api.spotify.com/v1/playlists/{playlistId}/tracks` — add tracks
- Returns: `{ playlistId, playlistUrl, trackCount, coverImageUrl }`

**`add_to_playlist`**
- Params: `{ playlistId: string, songs: string[] }` (songs are Spotify track URIs)
- Proxies to: `POST https://api.spotify.com/v1/playlists/{playlistId}/tracks`
- Returns: `{ playlistId, trackCount }`

### Authentication Flow

1. Extract NextAuth session from the request
2. Look up Spotify tokens in the `Account` table (`provider = 'spotify'`, `userId` from session)
3. If no token → return `{ error: "AUTH_REQUIRED" }` with HTTP 401
4. If `expires_at` is in the past → refresh the token (see below)
5. Attach `Authorization: Bearer {access_token}` to the Spotify API request
6. Forward the response

### Token Refresh

Spotify access tokens expire after 1 hour. The proxy checks `expires_at` before each call:

1. If expired: `POST https://accounts.spotify.com/api/token` with `grant_type=refresh_token`
2. Update `access_token`, `expires_at` in the `Account` table
3. Proceed with the refreshed token

If refresh fails (revoked, invalid), return `{ error: "AUTH_REQUIRED" }` so the user can re-authorize.

### Timeout

30-second fetch timeout on all outbound Spotify API calls. On timeout, return HTTP 504 with `{ error: "UPSTREAM_TIMEOUT" }`.

## 3. postMessage Protocol Extension

Two new message types for iframe-initiated data requests. These supplement the existing 7-type protocol and are used by `external_authenticated` plugins for interactive use (e.g., user searches within the iframe without going through the LLM).

### `API_REQUEST` (iframe → parent)

```json
{
  "type": "API_REQUEST",
  "requestId": "req_abc123",
  "payload": {
    "action": "search_songs",
    "params": { "query": "bohemian rhapsody" }
  }
}
```

`requestId` is a unique ID generated by the iframe to correlate with the response.

### `API_RESPONSE` (parent → iframe)

```json
{
  "type": "API_RESPONSE",
  "requestId": "req_abc123",
  "payload": {
    "data": { ... },
    "error": null
  }
}
```

On failure:

```json
{
  "type": "API_RESPONSE",
  "requestId": "req_abc123",
  "payload": {
    "data": null,
    "error": { "code": "AUTH_REQUIRED", "message": "Spotify account not connected" }
  }
}
```

### Routing

The parent (PluginBridge) intercepts `API_REQUEST` messages from the iframe, calls `/api/plugins/proxy/spotify` with the session cookie, and sends the result back as `API_RESPONSE`. The iframe never makes direct HTTP calls.

This pattern generalizes to any `external_authenticated` plugin — they send `API_REQUEST`, the parent proxies through the appropriate backend route, and returns `API_RESPONSE`.

### LLM-Initiated vs Iframe-Initiated

- **LLM-initiated:** User says "search for bohemian rhapsody" → LLM emits `search_songs` tool call → backend calls Spotify proxy → gets results → parent sends `INVOKE_TOOL` to iframe with results in the payload → iframe renders and sends `TASK_COMPLETE`
- **Iframe-initiated:** User types in the iframe's search box → iframe sends `API_REQUEST` → parent proxies → parent sends `API_RESPONSE` → iframe renders results (no `TASK_COMPLETE` needed, this is a UI interaction)

### Features

**Search panel:**
- Debounced search input (300ms) — sends `API_REQUEST` on each keystroke pause
- Results as a scrollable list with album art thumbnails (64x64), track name, artist, album, duration
- 30-second audio preview button per track — HTML5 `<audio>` element, toggles play/pause
- "Add" button adds track to the playlist builder below

**Playlist builder panel:**
- Editable playlist name input
- Drag-to-reorder tracks — HTML5 Drag and Drop API (no library needed for simple reorder)
- Remove button (×) per track
- Track count display
- "Create Playlist" button — sends `API_REQUEST` with `create_playlist` action

**Created playlist panel** (shown after creation):
- Playlist cover image from Spotify
- Playlist name + track count
- "Open in Spotify" link

**Status bar:**
- Shows connection state (connected / not connected)
- Shows error messages (rate limited, auth expired, etc.)

### Auth Error Handling

If `API_RESPONSE` contains `error.code === "AUTH_REQUIRED"`:
- Iframe shows a "Connect Spotify" message in the status bar
- Iframe sends `ERROR` with code `UPSTREAM_ERROR` and `message: "Spotify account not connected"`
- Platform renders a native "Connect Spotify" button in the chat UI (top-level NextAuth redirect)
- After OAuth completes, the iframe receives a fresh `INVOKE_TOOL` to retry

## 5. Platform Circuit Breaker

Applies to all plugins (chess, timeline, spotify). Uses the existing `failureCount` and `status` fields on `PluginRegistration`.

### Circuit Breaker Logic

**`server/lib/circuit-breaker.ts`**

Three functions:

- `recordFailure(appSlug)` — increments `failureCount` in the database. If `failureCount >= 3`, sets `status = 'unreliable'`.
- `recordSuccess(appSlug)` — resets `failureCount` to 0, sets `status = 'active'`.
- `isReliable(appSlug)` — returns `status === 'active'`.

### Integration Points

- **Plugin registry** (`GET /api/plugins`) — already filters by `status: 'active'`. Unreliable plugins are automatically excluded from LLM tool discovery.
- **Spotify proxy** — on Spotify API failure (5xx, timeout), calls `recordFailure('spotify')`. On success, calls `recordSuccess('spotify')`.
- **Plugin state route** — on `ERROR` postMessage with `recoverable: false`, the platform calls `recordFailure(appSlug)`.

### Reset Endpoint

**`PUT /api/plugins/[appSlug]/reset`**

Request:
```json
{ "apiKey": "sk_live_xyz789" }
```

Validates the `apiKey` against `PluginRegistration.apiKey`. If valid, resets `failureCount` to 0 and `status` to `'active'`.

Returns:
```json
{ "appSlug": "spotify", "status": "active", "failureCount": 0 }
```

## 7. Verification

- **OAuth flow:** Log in with credentials → click "Connect Spotify" → Spotify consent screen → redirected back → token stored in Account table
- **Search:** "Search for bohemian rhapsody" → LLM emits `search_songs` → proxy hits Spotify → iframe renders results with album art and preview buttons
- **Interactive search:** User types directly in iframe search box → `API_REQUEST` → proxy → `API_RESPONSE` → results rendered
- **Create playlist:** User builds playlist in iframe → clicks "Create Playlist" → `API_REQUEST` → proxy creates playlist → iframe shows playlist link
- **Token refresh:** Wait >1 hour → search again → proxy auto-refreshes token → works transparently
- **Auth required:** New user without Spotify connected → proxy returns `AUTH_REQUIRED` → chat UI shows "Connect Spotify" button
- **Circuit breaker:** Simulate 3 consecutive Spotify API failures → plugin flagged `unreliable` → excluded from LLM tool discovery → reset via API → plugin active again
- **State persistence:** Build a partial playlist → navigate away → return → `STATE_RESTORE` restores draft
